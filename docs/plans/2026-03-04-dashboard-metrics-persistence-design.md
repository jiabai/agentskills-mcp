# 概览统计持久化设计（方案A）

## 架构与数据模型
采用按时间桶聚合持久化的方案，满足审计与运营报表的可追溯性，同时控制写入量。新增统计表以「用户 + 小时桶」为主键，字段包含：user_id、bucket_start（UTC 小时起始）、total_count、success_count、failure_count、updated_at。每次 MCP tool 调用完成时对当前用户与当前小时桶进行计数递增（UPSERT），保证并发安全。概览接口 `/api/v1/dashboard/overview` 读取最近 24 个小时桶，按用户聚合成功率与次数，并返回 success_rate 与 success_rate_total。该方案跨重启与多实例一致，适合日报/周报/运营报表扩展。单次调用明细不持久化，以降低存储与查询成本。

## 数据流与异常处理
数据流从 tool 调用完成处写入统计：tool 根据请求上下文取得 user_id，写入聚合表。统计接口查询时先计算时间窗（now-24h 到 now），再聚合对应桶；若查询结果为空则返回 success_rate=0、success_rate_total=0。异常处理上，统计写入失败不影响 tool 返回，但要记录日志；可设置轻量重试或降级为内存计数（仅作为 debug，不进入产品指标）。为避免时间漂移，所有桶时间基准以 UTC 对齐到小时。

## 测试与验收
新增单元/集成测试覆盖：1) 无 Skill 时 success_rate=0 且 total=0。2) 生成模拟请求写入两小时桶，确认 24h 聚合正确。3) 排除 `/api/v1/auth` 与非业务路由的统计写入。4) 多次写入同一桶时计数累加正确。性能验收关注写入 QPS 与查询延迟，目标是在控制台刷新频率下保持毫秒级聚合查询。数据清理策略可按保留周期（如 30/90 天）对旧桶进行归档或删除。
