# 注册成功后交互设计（方案B）

## 目标与范围
本设计覆盖“创建账户”提交成功后的交互方式，选择方案B：注册成功后显示成功提示并自动跳转到 `/login`，由用户在登录页完成登录。该流程强调清晰反馈与可预期的跳转，避免无状态地停留在注册页导致用户误以为未完成注册。设计覆盖注册页的成功/失败状态提示、按钮可用性、跳转时机与文案一致性，不涉及后端接口改造，也不引入自动登录或Token下发。页面范围限定在 `/register` 与 `/login`，并遵循现有前端架构（Next.js App Router + shadcn/ui）。用户期望是：点击“创建账户”后获得明确成功反馈、知道下一步是登录，并能无缝进入登录页。

## 交互流程与状态
1) 用户填写用户名、邮箱、密码并提交。提交后按钮显示“正在创建...”，表单输入保持只读/禁用以避免重复提交。2) 注册成功：在表单下方展示成功提示文案“注册成功，请登录。”同时清空表单字段，避免用户误以为仍需再次提交。成功提示出现后启动短暂倒计时（建议 1.5–2s），到时自动跳转 `/login`。同时提供明确的行动引导：按钮或链接“立即登录”，可让用户手动提前跳转。3) 注册失败：保持当前输入内容，展示错误提示（例如“邮箱已注册”“用户名已存在”或其他后端错误），并允许用户修改后再次提交。失败时不触发跳转。4) 从登录页可通过“创建账户”链接返回注册页，形成闭环。

## 体验细节与异常处理
文案与状态要保持一致：成功提示使用主色强调，错误提示使用危险色；成功与错误不可同时展示，后到状态覆盖前一状态。自动跳转前应保证成功提示可见且可阅读，避免瞬间跳转导致“没看到就跳走”。如果用户在成功提示出现后手动点击“立即登录”，应立即跳转并清理倒计时，避免重复路由替换。无网络或服务器错误时，错误提示需保留且按钮恢复可用，避免用户卡死。由于方案B不自动登录，用户进入登录页后需自行输入凭证；登录页在成功登录后仍保持现有行为：保存Token并跳转 `/dashboard`。该方案兼容现有前端鉴权门控，确保未登录时访问控制台会被重定向到 `/login`。
